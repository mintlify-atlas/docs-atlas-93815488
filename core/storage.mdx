The storage subsystem provides a unified API for persisting settings across power cycles. It wraps platform-specific flash backends with write throttling and dual-core safety.

Source: `src/core/services/storage/storage.h`, `storage.c`, `flash.h`, and `flash.c`

## What gets persisted

The `flash_t` structure is exactly 256 bytes (one flash page) and holds:

```c
typedef struct {
    // Header (8 bytes)
    uint32_t magic;               // Validation magic (0x47435052 = "GCPR")
    uint32_t sequence;            // Monotonic sequence number (higher = newer, 0xFFFFFFFF = empty)

    // Global settings (4 bytes)
    uint8_t active_profile_index; // Currently selected profile (0=default, 1-4=custom)
    uint8_t usb_output_mode;      // USB device output mode (0=HID, 1=XboxOG, etc.)
    uint8_t wiimote_orient_mode;  // Wiimote orientation (0=Auto, 1=Horizontal, 2=Vertical)
    uint8_t custom_profile_count; // Number of custom profiles (0-4)

    uint8_t ble_output_mode;      // BLE output mode (0=Standard composite, 1=Xbox BLE)
    uint8_t reserved[19];         // Reserved for future global settings

    // Custom profiles (4 x 56 = 224 bytes)
    custom_profile_t profiles[CUSTOM_PROFILE_MAX_COUNT];
} flash_t;
```

<Note>
`sizeof(flash_t) == 256` is enforced at compile time with `_Static_assert`. Changing the struct layout breaks existing flash storage — increment the magic number if you need to invalidate old data.
</Note>

### Custom profiles

Each `custom_profile_t` (56 bytes) stores user-configured remapping and is created via the web configuration tool:

```c
typedef struct {
    char name[CUSTOM_PROFILE_NAME_LEN];                // 12 bytes, null-terminated
    uint8_t button_map[CUSTOM_PROFILE_BUTTON_COUNT];   // 18 bytes (one entry per button)
    // Button map values:
    //   0x00 = passthrough (no remap)
    //   0x01-0x12 = remap to JP_BUTTON_* (1-based)
    //   0xFF = disabled
    uint8_t left_stick_sens;   // 0-200 (100 = 1.0x, 50 = 0.5x, 200 = 2.0x)
    uint8_t right_stick_sens;  // 0-200
    uint8_t flags;             // Bit 0: swap sticks, Bit 1: invert LY, Bit 2: invert RY
    uint8_t socd_mode;         // 0=passthrough, 1=neutral, 2=up-priority, 3=last-win
    uint8_t reserved[22];
} custom_profile_t;
```

## Platform backends

| Platform | Backend | Source | Safety mechanism |
|----------|---------|--------|------------------|
| RP2040 | Raw flash, journaled ring buffer | `src/core/services/storage/flash.c` | `flash_safe_execute()` for dual-core coordination |
| ESP32-S3 | NVS (Non-Volatile Storage) | `esp/main/flash_esp32.c` | Mutex-protected by ESP-IDF |
| nRF52840 | Zephyr NVS | `nrf/src/flash_nrf.c` | Mutex-protected by Zephyr |

The API surface (`storage_init()`, `storage_task()`, `flash_get_settings()`, `flash_save()`) is identical across all three platforms.

## RP2040 flash layout

The RP2040 implementation uses journaled storage in two 4KB flash sectors near the end of flash:

```
Flash layout (from end):
  ... firmware code ...
  [Sector B: 4KB]  -- 16 x 256-byte journal slots (backup)
  [Sector A: 4KB]  -- 16 x 256-byte journal slots (primary)
  [BTstack: 8KB]   -- Bluetooth bond storage
  [RP2040 flash end]
```

### Journal design

Each 4KB sector holds 16 x 256-byte slots arranged as a ring buffer:

1. Each save writes to the next empty slot via page program (~1ms, no erase needed)
2. The `sequence` field identifies the newest entry (highest number wins)
3. When all 16 slots are full, the sector is erased (~45ms) and writing restarts from slot 0
4. Sector erase is deferred if Bluetooth is active to avoid latency spikes

This design minimizes flash wear. RP2040 flash supports roughly 100K erase cycles per sector — with a 5-second debounce and 16-slot journal, a sector erase happens at most every 80 seconds of continuous saves.

## Dual-core safety (RP2040)

Flash on RP2040 is memory-mapped and shared between both cores. A flash write while Core 1 is executing from flash causes a hard fault.

- `flash_safe_execute()` (from pico-sdk) coordinates between cores: pauses Core 1, disables interrupts, performs the flash operation, then resumes
- If `flash_safe_execute()` fails (e.g., Core 1 cannot be paused), the implementation falls back to a direct flash operation with interrupts disabled
- Core 1 code that runs during flash writes must be in RAM — use `__not_in_flash_func` for timing-critical Core 1 functions

<Warning>
On Pico 2 W (RP2350), Core 0's CYW43 driver periodically locks flash. Core 1 functions must be RAM-only (`__not_in_flash_func`). This restriction applies regardless of whether a flash write is in progress.
</Warning>

## Write throttling

Changes are debounced with a 5-second delay to prevent rapid profile cycling from wearing out flash:

1. `flash_save()` sets a pending flag and records the new settings
2. `flash_task()` (called every main loop iteration) checks the timer
3. After 5 seconds with no new save requests, the actual flash write occurs
4. `flash_save_now()` bypasses debouncing for immediate writes
5. `flash_save_force()` also bypasses the BT-active check (use before reboot)

## Bluetooth-aware writes

On RP2040 with Bluetooth active, sector erase causes audio/latency spikes in BT output. The flash system tracks BT connection state:

- Sector erase is deferred while BT is connected
- `flash_on_bt_disconnect()` — call this when BT disconnects; triggers any pending deferred write
- `flash_has_pending_write()` — returns true if a write is waiting for BT to become idle

## API

```c
// Initialization and main loop
void storage_init(void);   // Call once at boot (loads settings from flash)
void storage_task(void);   // Call every main loop iteration (handles debounced writes)

// Access settings
flash_t* flash_get_settings(void);              // Get runtime settings pointer
void flash_save(const flash_t* settings);        // Debounced save (5s delay)
void flash_save_now(const flash_t* settings);    // Immediate save
void flash_save_force(const flash_t* settings);  // Immediate, ignores BT state

// Profile helpers
uint8_t flash_get_active_profile_index(void);    // 0=default, 1-4=custom
void flash_set_active_profile_index(uint8_t index); // Switch profile, debounced save
uint8_t flash_get_total_profile_count(void);     // 1 default + custom_profile_count
void flash_cycle_profile_next(void);             // Next profile (wraps)
void flash_cycle_profile_prev(void);             // Previous profile (wraps)

// Active custom profile
const custom_profile_t* flash_get_active_custom_profile(void); // NULL for index 0

// BT-aware write coordination
void flash_on_bt_disconnect(void);   // Notify flash that BT is idle
bool flash_has_pending_write(void);  // Check if a write is waiting for BT idle

// Custom profile helpers
void custom_profile_init(custom_profile_t* profile, const char* name);
uint32_t custom_profile_apply_buttons(const custom_profile_t* profile, uint32_t buttons);
const custom_profile_t* flash_get_custom_profile(const flash_t* settings, uint8_t index);
```

## Profile persistence

Profile selections survive power cycles:

- `flash_set_active_profile_index()` writes the new index via the debounced save path
- At boot, `storage_init()` loads flash and restores `active_profile_index`
- Custom profiles (created in the web config tool) are stored directly in `flash_t.profiles[]`

## Factory reset

Factory reset clears all settings, profiles, and Bluetooth bonds and returns the device to firmware defaults. It can be triggered from the web configuration tool (Device Management → Factory Reset) or programmatically by writing a zeroed `flash_t` with `flash_save_force()` and rebooting.

After factory reset:
- `active_profile_index` returns to 0 (default passthrough)
- `custom_profile_count` returns to 0
- `usb_output_mode` returns to 0 (HID/SInput)
- All Bluetooth bonds are cleared (BTstack bond storage in the 8KB BTstack area)

## Platform-specific notes

<Tabs>
  <Tab title="ESP32-S3">
    Uses ESP-IDF NVS (Non-Volatile Storage), which manages wear leveling and journaling internally. The `flash_t` struct is serialized to NVS as a single blob under the key `"settings"`. No manual sector management or dual-core coordination is needed — ESP-IDF handles all of this with a mutex.

    BLE only (no Classic Bluetooth). The BT-aware write deferral logic (`flash_on_bt_disconnect`) still applies when a BLE connection is active.
  </Tab>
  <Tab title="nRF52840">
    Uses Zephyr NVS, which also handles wear leveling internally. The implementation is structurally identical to ESP32-S3 but uses Zephyr kernel primitives (mutex, work queue) instead of FreeRTOS.

    BTstack runs in its own Zephyr thread. Debug output goes to UART (not CDC/USB), so flash-related log messages appear on the UART console.
  </Tab>
  <Tab title="RP2040">
    Raw flash with a manual journaled ring buffer. Two 4KB sectors provide a primary and backup, giving 32 total journal slots before a full erase is needed.

    `flash_safe_execute()` from pico-sdk coordinates the dual-core flash access. Functions called from Core 1 during a potential write window must be placed in RAM with `__not_in_flash_func`.
  </Tab>
</Tabs>
