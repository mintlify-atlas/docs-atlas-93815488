---
title: "nRF52840 Platform"
description: "BLE-only platform for bt2usb and usb2usb apps. Uses nRF Connect SDK (Zephyr) with BTstack and TinyUSB to maximize shared code with other platforms."
---

Joypad OS supports nRF52840 as a build target for **bt2usb** (BLE to USB adapter) and **usb2usb** (USB host via MAX3421E to USB device). The nRF52840 has BLE and native USB, making it a compact standalone wireless controller adapter. Like the ESP32-S3 port, it uses the same shared Joypad OS core — the same BT HID drivers, router, profiles, and USB device output modes — running under Zephyr RTOS.

## Overview

| Feature | Pico W / Pico 2 W | ESP32-S3 | nRF52840 |
|---|---|---|---|
| Bluetooth | Classic BT + BLE | BLE only | BLE only |
| USB output | USB Device (RP2040) | USB OTG Device | USB Device |
| USB host | PIO-USB | No | Via MAX3421E (SPI) |
| Build system | pico-sdk / CMake | ESP-IDF / CMake | nRF Connect SDK / west / CMake |
| Firmware update | Drag-and-drop (.uf2) | Drag-and-drop (.uf2) via TinyUF2 | Drag-and-drop (.uf2) |

<Warning>
  nRF52840 supports BLE only — Classic Bluetooth is not enabled in this port. Controllers that pair exclusively over Classic BT (DualShock 3/4, DualSense, Switch Pro, Wiimote) will not work. Use a Pico W build for those controllers.
</Warning>

### Supported controllers (BLE)

| Controller | Status |
|---|---|
| Xbox One / Series (BLE mode) | Supported |
| 8BitDo controllers (BLE mode) | Supported |
| Switch 2 Pro (BLE) | Supported |
| Generic BLE HID gamepads | Supported |

## Supported boards

| Board | UF2 drive | UART pins | Supported apps | Notes |
|---|---|---|---|---|
| Seeed XIAO nRF52840 | `XIAO-SENSE` | TX=D6 (P1.11), RX=D7 (P1.12) | bt2usb | Default board; ultra-compact (0.84" × 0.70") |
| Adafruit Feather nRF52840 | `FTHR840BOOT` | TX=D1 (P0.25), RX=D0 (P0.24) | bt2usb, usb2usb | Feather form factor; MAX3421E FeatherWing for usb2usb |

Both boards use a UF2 bootloader — double-tap reset to enter bootloader mode and expose the USB drive.

## Prerequisites

nRF Connect SDK v3.1.0+ must be installed separately from the ARM toolchain used for RP2040 builds. The `make init-nrf` command handles the full workspace setup.

## One-time setup

<Steps>
  <Step title="Run the NCS installer">
    From the repo root:

    ```bash
    make init-nrf
    ```

    This sets up a nRF Connect SDK workspace using `west init` and `west update`. Requires Python 3 and pip. The SDK is installed in a `ncs/` directory adjacent to the repo.
  </Step>
  <Step title="Verify the setup">
    After `init-nrf` completes, confirm the build works:

    ```bash
    make bt2usb_seeed_xiao_nrf52840
    ```

    A successful build produces a `.uf2` file in `releases/`.
  </Step>
</Steps>

## Build commands

From the repo root:

```bash
# bt2usb — BLE to USB HID adapter
make bt2usb_seeed_xiao_nrf52840             # Build for XIAO
make bt2usb_adafruit_feather_nrf52840       # Build for Feather

# usb2usb — USB host (MAX3421E) to USB device (Feather only)
make usb2usb_feather_nrf52840               # Build for Feather

# Flash via UF2 bootloader
make flash-bt2usb_seeed_xiao_nrf52840       # Flash XIAO
make flash-bt2usb_adafruit_feather_nrf52840 # Flash Feather

# UART serial monitor (115200 baud)
make monitor-bt2usb_seeed_xiao_nrf52840     # Monitor XIAO
make monitor-bt2usb_adafruit_feather_nrf52840 # Monitor Feather
```

Or from the `nrf/` directory directly:

```bash
make build                                               # bt2usb for XIAO (default)
make build BOARD=adafruit_feather_nrf52840               # bt2usb for Feather
make build APP_TYPE=usb2usb BOARD=adafruit_feather_nrf52840  # usb2usb for Feather
make flash-uf2                                           # Flash via UF2 drive
make monitor                                             # UART serial monitor
```

Output: `releases/joypad_<commit>_<app>_<board>.uf2`

### Flashing

<Steps>
  <Step title="Enter bootloader mode">
    Double-tap the reset button. A USB drive appears: `XIAO-SENSE` for the XIAO nRF52840, or `FTHR840BOOT` for the Feather.
  </Step>
  <Step title="Flash via make">
    ```bash
    make flash-bt2usb_seeed_xiao_nrf52840
    ```

    This copies the `.uf2` to the mounted UF2 drive. The board reboots automatically into the new firmware.
  </Step>
  <Step title="Flash manually (alternative)">
    Copy the `.uf2` from `releases/` directly onto the USB drive. The board reboots when the copy completes.
  </Step>
</Steps>

A J-Link debug probe is also supported for flashing and debugging via SWD:

```bash
cd nrf && make flash-jlink
```

## Architecture

### Threading model

nRF52840 uses Zephyr RTOS with two threads:

```
Main Thread                  BTstack Thread
  USB device polling           BLE scanning / pairing
  LED status updates           HID report processing
  Button input                 Controller data → router
  Storage persistence
```

BTstack runs in its own Zephyr thread created with `k_thread_create`. All BLE operations happen in that thread. The main thread handles USB output, LED, and button input, yielding via `k_msleep(1)`.

### Key differences from RP2040

- **Zephyr RTOS** instead of bare-metal dual-core
- **Raw HCI passthrough** — Zephyr provides the BLE controller via `CONFIG_BT_HCI_RAW=y`; BTstack acts as the BLE host on top of that
- **TinyUSB owns USB** — Zephyr's USB device stack is disabled (`CONFIG_USB_DEVICE_STACK=n`); TinyUSB's `dcd_nrf5x.c` drives the USB hardware directly
- **NVS storage** — Flash persistence uses Zephyr NVS, shared between app settings and BTstack bond storage
- **MAX3421E for USB host** — For the usb2usb app on Feather, an external SPI-connected MAX3421E IC provides USB host capability

### nRF-specific files

| File | Purpose |
|---|---|
| `nrf/src/main.c` | Zephyr entry point |
| `nrf/src/flash_nrf.c` | NVS-based settings storage |
| `nrf/src/button_nrf.c` | Button stub (no user button on XIAO) |
| `nrf/src/btstack_config.h` | BLE-only BTstack config wrapper |
| `nrf/src/tusb_config_nrf.h` | nRF5x TinyUSB config |
| `nrf/prj.conf` | Zephyr Kconfig (`CONFIG_BT_HCI_RAW=y`, Zephyr USB disabled) |
| `nrf/boards/xiao_ble.overlay` | Disables the Zephyr USBD device node |
| `nrf/boards/xiao_ble.conf` | XIAO nRF52840 board-specific config |

## Debugging

### UART console

Since TinyUSB owns the USB peripheral, CDC serial is not available. UART is the primary debug output method. Connect a USB-to-UART adapter to the board's UART pins:

| Board | TX pin | RX pin | Baud rate |
|---|---|---|---|
| XIAO nRF52840 | D6 (P1.11) | D7 (P1.12) | 115200 |
| Feather nRF52840 | D1 (P0.25) | D0 (P0.24) | 115200 |

```bash
make monitor-bt2usb_seeed_xiao_nrf52840   # Auto-detects serial port
```

UART output works from boot, in fault handlers, and from ISR context — no debug probe required for basic diagnostics.

### Crash diagnostics

Crash dumps include register state and a stack trace. Common fault codes:

| Code | Meaning |
|---|---|
| `K_ERR_KERNEL_PANIC (reason=4)` | Software panic from `k_panic()` or `abort()` — typically a failed assertion in BTstack |
| `BusFault / HardFault` | Hardware fault — check for null pointer dereference or stack overflow |
| MPU fault | Stack overflow detected by MPU stack guard |

## Important nRF52840-specific notes

<Warning>
  **BTstack API calls must happen in the BTstack Zephyr thread.** Calling BTstack APIs from the main thread causes race conditions. Queue work to the BTstack thread instead.
</Warning>

<Warning>
  **Zephyr USB must be disabled.** The `prj.conf` sets `CONFIG_USB_DEVICE_STACK=n` and the board overlay sets `&usbd { status = "disabled"; }`. Removing these causes TinyUSB and Zephyr to fight over the USB hardware.
</Warning>

<Note>
  Classic Bluetooth APIs (`gap_inquiry_*`, `gap_set_class_of_device()`, `gap_discoverable_control()`) are not available in this port. These are guarded in shared code with `#ifndef BTSTACK_USE_NRF`.
</Note>
