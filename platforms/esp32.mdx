title: "ESP32-S3 Platform"
description: "BLE-only platform for the bt2usb app. Uses ESP-IDF with FreeRTOS for Bluetooth controller to USB HID output."
---

Joypad OS supports ESP32-S3 as a build target for the **bt2usb** app. The ESP32-S3 has both BLE and USB OTG, making it a compact standalone BLE-to-USB HID adapter. It uses the same shared Joypad OS core — the same BT HID drivers, router, profiles, and USB device output modes — with an ESP-IDF/FreeRTOS entry point instead of the pico-sdk.

## Overview

| Feature | Pico W / Pico 2 W | ESP32-S3 |
|---|---|---|
| Bluetooth | Classic BT + BLE | BLE only |
| USB output | USB Device (RP2040) | USB OTG Device |
| Controllers | All BT controllers | BLE controllers only |
| Build system | pico-sdk / CMake | ESP-IDF / CMake |
| Firmware update | Drag-and-drop (.uf2) | Drag-and-drop (.uf2) via TinyUF2 |

<Warning>
  ESP32-S3 supports BLE only — there is no Classic Bluetooth. Controllers that pair exclusively over Classic BT (DualShock 3/4, DualSense, Switch Pro, Wiimote) will not work. Use a Pico W build for those controllers.
</Warning>

### Supported controllers (BLE)

| Controller | Status |
|---|---|
| Xbox One / Series (BLE mode) | Supported |
| 8BitDo controllers (BLE mode) | Supported |
| Switch 2 Pro (BLE) | Supported |
| Generic BLE HID gamepads | Supported |

## Supported boards

| Board | Flash | Status | Notes |
|---|---|---|---|
| Seeed XIAO ESP32-S3 | 8MB | Tested | User LED on GPIO 21 (active low) |
| ESP32-S3-DevKitC | Varies | Should work | Untested |

The XIAO ESP32-S3 is the recommended board. It is the only ESP32 variant with both BLE and USB OTG hardware, which bt2usb requires.

## Prerequisites

ESP-IDF v6.0+ must be installed separately from the ARM toolchain used for RP2040 builds. The `make init-esp` command handles this automatically.

## One-time setup

<Steps>
  <Step title="Run the ESP-IDF installer">
    From the repo root:

    ```bash
    make init-esp
    ```

    This clones ESP-IDF v6.0 to `~/esp-idf` and installs the ESP32-S3 toolchain and Python dependencies. If `~/esp-idf` already exists, it skips the clone and only installs tools.
  </Step>
  <Step title="Verify the environment">
    The ESP-IDF environment is activated automatically by the Makefile targets. If you want to work directly from the `esp/` directory, source the environment script first:

    ```bash
    source esp/env.sh
    ```
  </Step>
</Steps>

## Build commands

From the repo root:

```bash
make bt2usb_esp32s3                 # Build
make uf2-bt2usb_esp32s3             # Build + generate .uf2
make flash-uf2-bt2usb_esp32s3       # Build + flash .uf2 via TinyUF2 drive
make flash-bt2usb_esp32s3           # Build + flash via esptool
make monitor-bt2usb_esp32s3         # UART serial monitor (Ctrl+] to exit)
```

Or from the `esp/` directory after sourcing `env.sh`:

```bash
make build      # Build
make uf2        # Build + generate .uf2
make flash      # Build + flash via esptool
make monitor    # UART serial monitor
```

Output: `releases/joypad_<commit>_bt2usb_esp32s3.uf2`

### Board-specific configuration

Board-specific sdkconfig overrides go in `esp/sdkconfig.board.<name>`. The XIAO ESP32-S3 config is at `esp/sdkconfig.board.xiao_esp32s3`. To add a custom board:

```bash
echo 'CONFIG_SOME_OPTION=y' > esp/sdkconfig.board.myboard
cd esp && make BOARD=myboard build
```

## TinyUF2 support

The ESP32-S3 build uses [Adafruit TinyUF2](https://github.com/adafruit/tinyuf2) as a bootloader, enabling the same drag-and-drop `.uf2` firmware updates as RP2040 boards.

### Flash layout

```
8MB Flash
────────────────────────────────────────────
0x000000  Bootloader (TinyUF2 2nd-stage)
0x008000  Partition table
0x009000  NVS (20KB)
0x00E000  OTA data (8KB)
0x010000  ota_0 (2MB)  — Joypad firmware
0x210000  ota_1 (2MB)  — unused
0x410000  factory (256KB)  — TinyUF2 mass storage app
0x450000  ffat (3.7MB)  — TinyUF2 virtual FAT filesystem
```

Normal boot: bootloader reads OTA data and boots `ota_0`. TinyUF2 mode: bootloader boots the factory partition and a USB drive appears. If `ota_0` is corrupted, the bootloader falls back to TinyUF2 automatically.

### First-time TinyUF2 setup

<Steps>
  <Step title="Download the TinyUF2 release">
    Download the release for your board from the [Adafruit TinyUF2 releases](https://github.com/adafruit/tinyuf2/releases) page. For the XIAO ESP32-S3, the file is named `tinyuf2-seeed_xiao_esp32s3-X.XX.X.zip`.
  </Step>
  <Step title="Flash combined.bin">
    Extract the zip and flash using esptool:

    ```bash
    esptool.py --chip esp32s3 -b 921600 write_flash 0x0 combined.bin
    ```
  </Step>
  <Step title="Verify">
    The device reboots into TinyUF2 and a USB drive named `XIAOS3BOOT` appears. You can now drop a `.uf2` file onto the drive or run `make flash-uf2-bt2usb_esp32s3`.
  </Step>
</Steps>

After this one-time step, all future firmware updates are drag-and-drop.

### Entering TinyUF2 mode

| Method | How |
|---|---|
| Double-tap reset | Press the reset button twice within 500ms |
| CDC command | Connect to the USB serial port and send `BOOTSEL` |
| Recovery | If the app partition is corrupted, TinyUF2 boots automatically |

## Architecture

### Threading model

ESP32-S3 uses FreeRTOS with two tasks:

```
Main Task                    BTstack Task
  USB device polling           BLE scanning / pairing
  LED status updates           HID report processing
  Button input                 Controller data → router
  Storage persistence
```

BTstack runs in its own FreeRTOS task. All BLE operations — scanning, pairing, HID data — happen in that task. The main task handles USB output, LED, and button input.

### Shared code

The ESP32-S3 build compiles the same shared source files as the Pico W build:

- `src/core/` — Router, profiles, players, hotkeys, storage, LEDs
- `src/usb/usbd/` — USB device output modes (HID, XInput, PS3, PS4, Switch)
- `src/bt/` — All BT HID drivers and BTstack host integration

Platform-specific code is abstracted through `src/platform/platform.h`.

### ESP32-specific files

| File | Purpose |
|---|---|
| `esp/main/main.c` | FreeRTOS entry point |
| `esp/main/flash_esp32.c` | NVS-based settings storage |
| `esp/main/button_esp32.c` | GPIO button driver |
| `esp/main/btstack_config.h` | BLE-only BTstack configuration |
| `esp/main/tusb_config_esp32.h` | TinyUSB device configuration |
| `esp/partitions_uf2.csv` | Partition table matching TinyUF2 layout |
| `src/platform/esp32/platform_esp32.c` | Platform HAL (time, reboot, double-tap reset) |
| `src/bt/transport/bt_transport_esp32.c` | BLE transport layer |

## Important ESP32-specific notes

<Warning>
  **`tud_task()` blocks forever on FreeRTOS.** On ESP32, `tud_task()` is equivalent to `tud_task_ext(UINT32_MAX, false)` and will never return. Always use `tud_task_ext(1, false)` in the main task loop.
</Warning>

<Warning>
  **BTstack API calls must happen in the BTstack FreeRTOS task.** Calling BTstack APIs from the main task causes race conditions. Queue work to the BTstack task instead.
</Warning>

<Note>
  Classic Bluetooth APIs (`gap_inquiry_*`, `gap_set_class_of_device()`, `gap_discoverable_control()`) are not available on ESP32-S3. These are guarded in shared code with `#ifndef BTSTACK_USE_ESP32`.
</Note>
