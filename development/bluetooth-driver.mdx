---
title: "Adding a Bluetooth Device Driver"
description: "How to write a Bluetooth HID device driver using the bthid_driver_t interface and register it in the BTHID registry."
---

Bluetooth device drivers use the same conceptual pattern as USB device drivers — match a device, initialize it, parse incoming HID reports, normalize to `input_event_t`, and call `router_submit_input()` — but they are implemented against the `bthid_driver_t` interface instead of `DeviceInterface`, and they receive data from BTstack instead of TinyUSB.

## File location

```
src/bt/bthid/devices/vendors/<vendor>/<device>.c
src/bt/bthid/devices/vendors/<vendor>/<device>.h
```

Existing vendor directories: `sony/`, `nintendo/`, `google/`, `microsoft/`.

## The bthid_driver_t struct

Defined in `src/bt/bthid/bthid.h`:

```c
typedef struct {
    const char* name;

    // Check if this driver handles a device
    // Priority: VID/PID > device name > Class of Device (COD)
    bool (*match)(const char* device_name, const uint8_t* class_of_device,
                  uint16_t vendor_id, uint16_t product_id, bool is_ble);

    // Initialize driver for a connected device
    bool (*init)(bthid_device_t* device);

    // Process an incoming HID report
    void (*process_report)(bthid_device_t* device, const uint8_t* data, uint16_t len);

    // Periodic task for output reports (rumble, LEDs)
    void (*task)(bthid_device_t* device);

    // Device disconnected
    void (*disconnect)(bthid_device_t* device);
} bthid_driver_t;
```

The `bthid_device_t` context struct passed to each callback carries device state:

```c
typedef struct {
    bool active;
    uint8_t conn_index;        // Transport connection index (use as dev_addr)
    uint8_t bd_addr[6];        // Bluetooth device address
    char name[48];             // Device name from SDP/advertising data
    uint16_t vendor_id;        // USB VID from SDP manufacturer data
    uint16_t product_id;       // USB PID from SDP manufacturer data
    bool is_ble;               // true if BLE, false if Classic BT
    const void* driver;        // Pointer to this driver struct
    void* driver_data;         // Driver-specific private data
} bthid_device_t;
```

## Implementing the five functions

### match

Determines whether this driver handles a given device. The BTHID registry calls `match()` in registration order — first match wins. Prefer VID/PID matching when SDP data is available. Fall back to name-based matching for devices that do not expose VID/PID.

```c
static bool acme_bt_match(const char* device_name,
                           const uint8_t* class_of_device,
                           uint16_t vendor_id, uint16_t product_id,
                           bool is_ble)
{
    // Highest priority: VID/PID match
    if (vendor_id == 0x1234 && product_id == 0x5678) {
        return true;
    }

    // Fallback: name-based match
    if (device_name && strstr(device_name, "ACME Controller") != NULL) {
        return true;
    }

    return false;
}
```

<Note>
  In BLE, HCI connection handle `0x0000` is valid. Use `HCI_CON_HANDLE_INVALID` (`0xFFFF`) as a sentinel value — never `0` — when checking for an invalid or unassigned handle.
</Note>

### init

Called once when the device connects and the driver is matched. Allocate a slot in your driver's private state array and store a pointer in `device->driver_data`. The `bthid_device_t*` pointer is valid for the lifetime of the connection.

```c
typedef struct {
    bool initialized;
    uint8_t rumble_left;
    uint8_t rumble_right;
} acme_bt_data_t;

static acme_bt_data_t acme_data[BTHID_MAX_DEVICES];

static bool acme_bt_init(bthid_device_t* device)
{
    for (int i = 0; i < BTHID_MAX_DEVICES; i++) {
        if (!acme_data[i].initialized) {
            acme_data[i].initialized = true;
            acme_data[i].rumble_left = 0;
            acme_data[i].rumble_right = 0;
            device->driver_data = &acme_data[i];
            printf("[acme_bt] init conn_index=%d is_ble=%d\n",
                   device->conn_index, device->is_ble);
            return true;
        }
    }
    return false;  // No free slots
}
```

### process_report

Receives the raw HID report bytes from BTstack. Parse the report, build an `input_event_t`, and call `router_submit_input()`. BT reports often have a 1-byte report ID prefix or (for enhanced DS4 reports) a 3-byte header — account for the offset before casting to your report struct.

```c
static void acme_bt_process_report(bthid_device_t* device,
                                    const uint8_t* data, uint16_t len)
{
    if (len < 1) return;

    // Skip report ID byte (0x01 for standard input reports)
    const uint8_t* report_data = data + 1;
    uint16_t report_len = len - 1;

    if (report_len < sizeof(acme_bt_report_t)) return;

    const acme_bt_report_t* rpt = (const acme_bt_report_t*)report_data;

    uint32_t buttons = 0;
    if (rpt->cross)   buttons |= JP_BUTTON_B1;
    if (rpt->circle)  buttons |= JP_BUTTON_B2;
    if (rpt->square)  buttons |= JP_BUTTON_B3;
    if (rpt->tri)     buttons |= JP_BUTTON_B4;
    if (rpt->l1)      buttons |= JP_BUTTON_L1;
    if (rpt->r1)      buttons |= JP_BUTTON_R1;
    if (rpt->l2_btn)  buttons |= JP_BUTTON_L2;
    if (rpt->r2_btn)  buttons |= JP_BUTTON_R2;
    if (rpt->select)  buttons |= JP_BUTTON_S1;
    if (rpt->start)   buttons |= JP_BUTTON_S2;
    if (rpt->l3)      buttons |= JP_BUTTON_L3;
    if (rpt->r3)      buttons |= JP_BUTTON_R3;
    if (rpt->home)    buttons |= JP_BUTTON_A1;

    // D-pad from hat switch
    if (rpt->dpad == 0 || rpt->dpad == 1 || rpt->dpad == 7) buttons |= JP_BUTTON_DU;
    if (rpt->dpad >= 1 && rpt->dpad <= 3)                   buttons |= JP_BUTTON_DR;
    if (rpt->dpad >= 3 && rpt->dpad <= 5)                   buttons |= JP_BUTTON_DD;
    if (rpt->dpad >= 5 && rpt->dpad <= 7)                   buttons |= JP_BUTTON_DL;

    input_event_t event;
    init_input_event(&event);
    event.dev_addr = device->conn_index;  // Use conn_index as device address
    event.instance = 0;
    event.type = INPUT_TYPE_GAMEPAD;
    event.transport = device->is_ble ? INPUT_TRANSPORT_BT_BLE
                                     : INPUT_TRANSPORT_BT_CLASSIC;
    event.buttons = buttons;
    event.analog[ANALOG_LX] = rpt->lx;
    event.analog[ANALOG_LY] = rpt->ly;  // HID convention, no inversion needed
    event.analog[ANALOG_RX] = rpt->rx;
    event.analog[ANALOG_RY] = rpt->ry;
    event.analog[ANALOG_L2] = rpt->l2_trigger;
    event.analog[ANALOG_R2] = rpt->r2_trigger;

    router_submit_input(&event);
}
```

### task

Called periodically from the BTHID task loop (Core 0). Use this to send output reports — rumble, LED color, player LED index — when the feedback state changes. Use `bthid_send_output_report()` to send data back to the controller:

```c
static void acme_bt_task(bthid_device_t* device)
{
    acme_bt_data_t* acme = (acme_bt_data_t*)device->driver_data;
    if (!acme) return;

    // Check if feedback state changed
    feedback_state_t* fb = feedback_get_state(device->conn_index);
    if (!fb || !fb->rumble_dirty) return;

    uint8_t output[4] = {
        0x01,                  // Report ID
        fb->rumble.right,      // High-frequency motor
        fb->rumble.left,       // Low-frequency motor
        0x00,
    };
    bthid_send_output_report(device->conn_index, 0x01, output, sizeof(output));
    feedback_clear_dirty(device->conn_index);
}
```

### disconnect

Called when the device disconnects. Release the private data slot.

```c
static void acme_bt_disconnect(bthid_device_t* device)
{
    acme_bt_data_t* acme = (acme_bt_data_t*)device->driver_data;
    if (acme) {
        acme->initialized = false;
        device->driver_data = NULL;
    }
    printf("[acme_bt] disconnected\n");
}
```

## Registering the driver

Add your driver to `src/bt/bthid/bthid_registry.c`:

<Steps>
  <Step title="Add the include">
    ```c
    #include "devices/vendors/acme/acme_bt.h"
    ```
  </Step>

  <Step title="Add a register function in your header">
    ```c
    // acme_bt.h
    void acme_bt_register(void);
    ```

    ```c
    // acme_bt.c
    static const bthid_driver_t acme_bt_driver = {
        .name           = "ACME BT Gamepad",
        .match          = acme_bt_match,
        .init           = acme_bt_init,
        .process_report = acme_bt_process_report,
        .task           = acme_bt_task,
        .disconnect     = acme_bt_disconnect,
    };

    void acme_bt_register(void)
    {
        bthid_register_driver(&acme_bt_driver);
    }
    ```
  </Step>

  <Step title="Call the register function in bthid_registry_init()">
    ```c
    void bthid_registry_init(void)
    {
        bthid_init();

        // Vendor-specific drivers (registered in priority order)
        ds3_bt_register();
        ds4_bt_register();
        ds5_bt_register();
        acme_bt_register();  // Add your driver here

        // Generic fallback (always last)
        bthid_gamepad_register();
    }
    ```
  </Step>
</Steps>

## BT-specific considerations

<AccordionGroup>
  <Accordion title="BLE vs Classic BT">
    BLE-only platforms (ESP32-S3, nRF52840) cannot connect Classic BT controllers. Check `device->is_ble` in your `match()` and `init()` functions if your driver only supports one transport.

    Classic BT APIs must be guarded to avoid compile errors on BLE-only builds:

    ```c
    #ifndef BTSTACK_USE_ESP32
    #ifndef BTSTACK_USE_NRF
    gap_inquiry_start(INQUIRY_DURATION);
    #endif
    #endif
    ```
  </Accordion>

  <Accordion title="BTstack threading">
    All BTstack API calls — including `bthid_send_output_report()` — must be made from the BTstack task or thread. On RP2040, this is Core 0's cooperative event loop. On ESP32 and nRF52840, it is a dedicated FreeRTOS task or Zephyr thread respectively. Never call BTstack APIs from Core 1 or from an interrupt handler.
  </Accordion>

  <Accordion title="Report ID offsets">
    BT HID reports do not have a uniform offset. Different devices use different prefixes:
    - Standard input reports typically start with a 1-byte report ID (`data[0]`).
    - DS4 enhanced BT reports (0x11) have a 3-byte header before the gamepad data.
    - Check your controller's BT HID descriptor or protocol documentation before assuming an offset.
  </Accordion>

  <Accordion title="Output report buffers">
    BTstack's `hid_host_send_set_report()` stores a pointer to your output buffer for a deferred L2CAP send. The buffer must remain valid until the `CAN_SEND_NOW` callback fires. Use a `static` buffer inside the driver, as done in the DS4 BT driver (`src/bt/bthid/devices/vendors/sony/ds4_bt.c`).
  </Accordion>
</AccordionGroup>
